## 一、 数据库与 SQL 相关问题

### Q1: 你们的数据库设计符合第几范式？有没有进行反规范化设计？
*   **答辩得分点**：三范式 (3NF)、传递依赖、反规范化 (Denormalization) 的权衡。
*   **回答话术**：
    > “我们的核心业务表（如 `employees`、`departments`、`payroll`、`leave_requests`）均**严格符合第三范式 (3NF)**。每个非主属性都直接依赖于主键，消除了部分依赖和传递依赖，从而最大程度避免了数据冗余和插入、删除、修改异常。
    > 
    > 但是，我们在 **`audit_logs`（操作审计日志表）** 中进行了**合理的反规范化设计**：该表除了存储 `emp_id`，还冗余存储了 `emp_name` 字段。
    > **原因有二**：一是审计日志是高频写入且需要永久保留的历史存证。如果只存 ID，每次展示日志都要与员工表进行 `JOIN` 查询，在日志量大时会严重消耗数据库性能；二是如果未来某个员工被删除或更改了姓名，审计日志仍需真实还原‘操作发生那一刻的员工姓名’，因此在此处冗余姓名不仅提升了查询效率，也保证了审计数据的历史真实性。”

### Q2: 什么是数据库事务 (Transaction)？你们在系统里哪里用到了事务？怎么保证 ACID 特性的？
*   **答辩得分点**：ACID 原理、Qt 中事务的 API 调用、一键核算工资业务。
*   **回答话术**：
    > “事务是作为一个逻辑工作单元执行的一系列操作，具有原子性 (A)、一致性 (C)、隔离性 (I) 和持久性 (D)。
    > 
    > 我们的系统中，**一键核算全员薪资**（对应 [PayrollTab::calculate](file:///c:/Users/17760/Desktop/HRMS/src/tabs/PayrollTab.cpp#L84)）业务必须使用事务。核算过程涉及读取所有员工考勤、绩效、请假数据，计算每一笔工资，并批量插入到 `payroll` 表中。
    > 
    > 在代码中，我们通过 Qt 的 `db.transaction()` 开启事务，如果所有员工的工资计算与写入均成功，则调用 `db.commit()` 提交；若中间任何一名员工的计算或插入抛出 SQL 异常，则在 `catch` 或错误处理分支中调用 `db.rollback()` 回滚所有已写入的数据。这样可以**彻底避免‘部分员工发了工资，部分员工没发’的数据不一致灾难**，完美保障了事务的原子性和一致性。”

### Q3: 为什么要设置外键约束？我们在代码中是如何优雅处理外键完整性约束的？
*   **答辩得分点**：参照完整性约束 (Referential Integrity)、防呆设计、Bug 规避。
*   **回答话术**：
    > “外键约束是为了保证数据库的**参照完整性**，防止出现孤立或无效的关联数据。例如 `notifications` 表中的 `emp_id` 外键关联了 `employees` 表的 `id`，防止向不存在的员工发送通知。
    > 
    > 在开发过程中，我们发现一个关于外键的隐蔽 Bug：在提交请假审批时，系统需要通知管理员，原代码直接在 `notifications` 中插入 `emp_id = 0` 表示发送给管理员，但这在开启外键约束的数据库中会直接因为‘不存在 ID 为 0 的员工’而触发外键冲突，导致系统插入失败甚至崩溃。
    > 
    > **我们的解决方案**：在 [MainWindow::notifyUser](file:///c:/Users/17760/Desktop/HRMS/src/ui/MainWindow.cpp#L143) 函数中进行了业务层拦截。如果传入的接收者 ID 为 0，则自动调用 `notifyAdmins()`，通过子查询动态找出所有管理员（角色为 `admin`）的 `emp_id`，并分别为他们插入通知。这一设计在业务层完成了容错，既严格遵守了数据库的外键约束，又实现了‘向全体管理员发通知’的业务诉求。”

### Q4: 很多 SQL 驱动下 `QSqlQuery::size()` 返回 `-1`，你们是怎么发现并解决这个兼容性问题的？
*   **答辩得分点**：ODBC 驱动机制、Qt 跨平台兼容、`QSqlQuery::next()` 的标准写法。
*   **回答话术**：
    > “在系统运行于 Windows + ODBC 驱动（如 QODBC 连接 MySQL）时，由于 ODBC 驱动默认采用**向前只读（Forward-Only）**游标，`QSqlQuery::size()` 无法在不遍历整个结果集的情况下获取总行数，因此始终返回 `-1`。
    > 
    > 原系统在进行数据库表结构动态迁移（例如检查 `payroll` 表中是否已存在新增的五险一金列，对应 [PayrollTab::initializeSchema](file:///c:/Users/17760/Desktop/HRMS/src/tabs/PayrollTab.cpp#L35)）时，使用了 `if (q.size() == 0)` 来判断字段是否存在。这在 QODBC 驱动下会导致条件永远成立，从而重复执行 `ALTER TABLE ADD COLUMN` 触发数据库报错。
    > 
    > **我们的优化**：我们将其重构为了标准的 `if (!q.next())`。通过尝试移动游标到第一行来判断结果集是否为空，此方法不受底层数据库驱动 and 游标类型的限制，完美解决了跨平台与不同 SQL 驱动下的兼容性隐患。”

### Q4b: 你们的可视化组织架构图是动态加载的，对于深层级部门，频繁的数据库递归查询是否会拖慢系统？你们是怎么优化的？
*   **答辩得分点**：一次性平摊查询（Flat Query）、内存树重建、O(N) 复杂度。
*   **回答话术**：
    > “在传统的递归层级查询中，通常会为每个节点执行一次 `SELECT` 语句（即 N+1 查询问题），如果部门层级很深或数量很多，会产生几十次网络 I/O，严重拖慢系统。
    > 
    > **我们的优化**：我们在 [OrgChartView::refresh](file:///c:/Users/17760/Desktop/HRMS/src/widgets/OrgChartView.cpp#L42) 中采用了**单次平铺查询（Flat Query）与内存树重建**的优化方案。我们仅执行一条简单的 SQL：`SELECT dept_id, dept_name, parent_id, manager_id FROM departments`，一次性将所有部门数据载入内存，然后利用 `QMap<int, OrgNode*>` 在内存中以 $O(N)$ 的时间复杂度快速重建出父子指针树状结构。
    > 
    > 这种设计将数据库压力降到了最低，即使公司有数千个部门，动态生成架构图的时间也仅需不到 1 毫秒，实现了极致的响应速度。”

## 二、 面向对象与 Qt 框架相关问题

### Q5: 项目中用到了哪些设计模式？请结合具体代码说明。
*   **答辩得分点**：单例模式 (Singleton)、观察者模式 (Observer)、MVC 模式。
*   **回答话术**：
    > “我们的系统应用了多种经典的面向对象设计模式，以实现解耦和高内聚：
    > 
    > 1.  **单例模式 (Singleton Pattern)**：
    >     全局事件总线 [GlobalEvents](file:///c:/Users/17760/Desktop/HRMS/src/core/GlobalEvents.h) 采用了单例模式。其构造函数设为私有，通过静态方法 `instance()` 获取唯一实例。这确保了整个客户端生命周期内只有一个事件分发中心。
    > 2.  **观察者模式 (Observer Pattern)**：
    >     结合了 Qt 的**信号与槽（Signal & Slot）**机制。各 Tab 页面（如 `DashboardTab`、`AuditTab`）在初始化时作为观察者，通过 `connect` 订阅了单例 `GlobalEvents` 的信号（如 `dataChanged()` 数据变更、`auditRefresh()` 审计刷新）。当 `EmployeeTab` 保存修改或发生审批时，事件源触发信号，所有订阅的 Tab 自动更新数据。这实现了界面模块间的松耦合。
    > 3.  **MVC 模式 (Model-View-Controller)**：
    >     Qt 的表格展现采用了 MVC 架构。`QSqlTableModel` 作为 **Model** 封装了与 MySQL 交互的数据源；`QTableView` 作为 **View** 负责界面的网格渲染；而 `MainWindow` 和各个 Tab 的槽函数则扮演 **Controller**，处理用户点击事件并协调 Model 更新。这种分离便于代码的单独测试与维护。”

### Q6: 什么是 Qt 的信号与槽机制？它的底层原理是什么？
*   **答辩得分点**：元对象编译器 (MOC)、事件循环、松耦合通信。
*   **回答话术**：
    > “信号与槽是 Qt 的核心机制，用于对象间的通信。
    > *   **信号 (Signal)**：当对象的某个状态发生改变时被发射（`emit`），它只是一个函数声明，不需要实现。
    > *   **槽 (Slot)**：是普通的成员函数，可以被连接到信号上，当信号发射时，槽函数会被自动调用。
    > 
    > 底层原理：Qt 通过**元对象编译器 (MOC, Meta-Object Compiler)** 在编译前对包含 `Q_OBJECT` 宏的头文件进行预处理，生成额外的 C++ 源文件（如 `moc_xxx.cpp`），其中包含了类的元信息和信号发射的查找表。当发射信号时，Qt 的元对象系统会通过连接记录，在事件循环中查找到所有关联的槽函数并依次调用。这种机制用安全、动态的查表调用代替了 C++ 传统的硬编码回调函数，极大地降低了类与类之间的耦合度。”

### Q6b: 你们的可视化组织架构图是如何使用 Qt 绘图框架实现的？如何解决画布平移、缩放以及高精度反走样的渲染瓶颈？
*   **答辩得分点**：Qt 图形视图框架 (Graphics View Framework)、矩阵变换 (QTransform)、反走样优化。
*   **回答话术**：
    > “我们没有采用第三方网页渲染，而是完全基于 **Qt 图形视图框架 (Graphics View Framework)** 开发了 `OrgChartView` 组件。
    > - **渲染管理**：使用 `QGraphicsScene` 容纳所有的部门节点卡片和直角连线；使用 `QGraphicsView` 作为视口提供滚动和交互。
    > - **缩放与平移**：开启了 `QGraphicsView::ScrollHandDrag` 拖拽模式，让用户可以直接用鼠标左键拖拽画布进行平移；通过重写 `wheelEvent` 并判定 `Ctrl` 键修饰符，通过 `scale(f, f)` 矩阵缩放实现平滑的无极缩放。
    > - **性能与高画质**：为了确保文字和圆角在大比例缩放时依然清晰，我们启用了 `QPainter::Antialiasing`（线条反走样）和 `QPainter::TextAntialiasing`（文本反走样）。同时，连线等静态元素绘制在底层（Z-value 为负值），避免重绘遮挡，保证了在低配电脑上也能达到 60 帧的流畅交互。”

## 三、 安全性与系统健壮性问题

### Q7: 用户的密码在数据库中是明文存储的吗？如何防范 SQL 注入？
*   **答辩得分点**：SHA-256 加密、哈希加盐（Salting）思想、参数化查询 (Parameterized Queries)。
*   **回答话术**：
    > “**绝对不是明文存储**。为了保障员工隐私和数据安全，我们采用 **SHA-256 安全哈希算法** 对密码进行单向加密。数据库的 `employees` 表中的 `password` 字段仅存储哈希值，登录验证时将输入密码以同样算法计算哈希并与数据库对比，即使数据库泄露也无法逆推出原始密码。
    > 
    > **防范 SQL 注入**：我们系统中的所有 SQL 操作，凡是涉及用户输入的（如登录框的用户名、搜索框的关键字、请假申请的文本），**一律使用参数化查询（即占位符 `?` 或冒号占位符，结合 `QSqlQuery::prepare` 和 `addBindValue`）**。这确保了 SQL 语句的语法树在执行前已由编译引擎确定，用户输入仅被当作纯字面量参数处理，彻底杜绝了 SQL 注入攻击的可能。”

### Q8: 客户端的数据库连接密码是如何保存的？之前遇到过什么问题？你们是怎么解决的？
*   **答辩得分点**：防呆设计、配置文件加密、Base64 冲突解决。
*   **回答话术**：
    > “客户端的数据库连接参数（如主机、端口、用户名、密码）保存在本地的 `config.ini` 文件中。为了防止明文密码被直接窃看，系统在读取密码时会使用 Base64 解混淆。
    > 
    > **曾遇到的问题**：原系统在‘服务器设置对话框’保存密码时直接以明文保存。这导致如果用户设置的密码刚好全部由 Base64 字母表中的字符组成（例如 `admin1`），系统在下次启动时会误判定该密码为 Base64 编码并进行解码，从而导致解密出乱码、连接数据库失败。
    > 
    > **我们的优化**：我们修改了 [ServerSettingsDialog::onSave](file:///c:/Users/17760/Desktop/HRMS/src/ui/ServerSettingsDialog.cpp#L212)，在保存密码时**统一强制进行 Base64 编码存盘**，确保写入配置文件的永远是加密混淆串。这样在系统启动读取时，无论是何种密码，均能通过统一的 Base64 解码恢复出正确密码，彻底解决了这个导致自动连接失效的致命 Bug。”

### Q9: 如果网络中断或 MySQL 数据库宕机，你的客户端会直接崩溃吗？如何处理的？
*   **答辩得分点**：优雅退化、防崩溃、异常捕获、重试机制。
*   **回答话术**：
    > “不会闪退或崩溃，这是我们系统在**健壮性（Robustness）**上的一个设计亮点。
    > 
    > 在客户端启动时，`main.cpp` 会首先尝试读取 `config.ini` 并建立数据库连接。如果连接失败，程序并不会崩溃，而是会**捕获错误信息并优雅地弹出‘服务器设置’界面**，提示用户‘数据库连接失败，请检查网络或配置’。
    > 
    > 在配置成功并测试通过后，系统才会允许进入登录界面。另外，在所有高频数据提交的槽函数中，我们都加入了对 `QSqlQuery::exec()` 返回值的安全校验。如果执行失败，会弹出 `QMessageBox::critical` 提示具体错误并回滚当前操作，绝对不会因为底层数据库的断开导致客户端发生空指针或段错误闪退。”

### Q9b: 如果员工忘记了密码，该如何自主找回？如何确保该找回流程的安全、防止恶意的爆破和冒领？
*   **答辩得分点**：多维交叉校验（工号+姓名+手机）、单项哈希重置、操作审计可追溯性。
*   **回答话术**：
    > “我们在登录界面新增了**忘记密码**的自主找回流，并设计了严密的安全机制来保障身份验证的可靠性：
    > 
    > 1.  **两阶段身份校验向导**：
    >     用户点击‘忘记密码？’后会弹出向导对话框 [RecoverPasswordDialog](file:///c:/Users/17760/Desktop/HRMS/src/ui/RecoverPasswordDialog.cpp)。第一阶段要求用户输入**工号、姓名、预留手机号**。系统会将这三项作为参数绑定，在 `employees` 表中检索是否存在完全一致且状态为‘在职’的员工记录。这防止了黑客仅凭姓名或手机号进行的批量猜测。
    > 2.  **单向哈希重置**：
    >     身份匹配成功后进入第二阶段，用户可设置新密码。新密码以 **SHA-256 单向不可逆哈希**形式存储，确保了密码库的密文安全性。
    > 3.  **审计日志可追溯性**：
    >     为防范恶意冒领和违规操作，密码重置成功后，系统会在后台静默向 `audit_logs` 表中插入一条审计流水，记录操作人、操作类型为‘找回密码’、详细描述为‘通过身份信息自主重置密码’。这样，安全审计人员能对每一笔自助密码找回操作进行源头追溯。”

## 四、 软件工程与项目实现问题

### Q10: 介绍一下你们系统的用户权限隔离（RBAC）是如何实现的？
*   **答辩得分点**：基于角色的权限控制 (RBAC)、动态数据库驱动、细粒度原子权限、缓存与实时更新、防御性安全隔离。
*   **回答话术**：
    > “我们实现了一套完全由**数据库驱动且支持自定义的动态 RBAC (Role-Based Access Control) 权限系统**。相较于常见的硬编码角色设计，我们的系统更加灵活、安全：
    > 
    > 1.  **数据库权限表设计**：
    >     系统在底层设计了 `roles`（角色表）、`permissions`（权限定义表）和 `role_permissions`（角色权限映射表）。系统预置了 17 项细粒度的原子权限（如 `view_dashboard` 查看大盘、`manage_employees` 员工管理、`approve_leave` 审批请假、`calculate_payroll` 核算薪酬等）。管理员可以在‘🔑 权限管理’页面动态添加自定义角色（如 `HR`、`部门主管`）并通过勾选复选框直接在表级更新角色对应的权限集。
    > 2.  **会话级缓存与实时刷新 (Session Manager)**：
    >     当用户登录成功后，[SessionManager](file:///c:/Users/17760/Desktop/HRMS/src/core/SessionManager.h) 统一从 `role_permissions` 加载当前角色所拥有的全部权限 Key 并缓存至内存 `QSet` 中。界面各模块通过 `SessionManager::instance()->hasPermission("perm_key")` 动态构建主页面。例如，如果某角色没有特定的管理权限，主窗口在加载时就会动态过滤该侧边栏或隐藏对应的子标签页、审批按钮等。如果管理员在后台修改了当前用户的角色权限，系统会立即触发 reload 并实时应用新权限，无需重启客户端。
    > 3.  **多层数据级越权拦截**：
    >     除了 UI 层的动态过滤，我们在业务数据模型层也进行了严格的安全限制。在加载请假和薪酬表格数据时，系统先进行权限检查：如果是拥有审批/计算权限的角色，SQL 允许拉取全量数据；如果是普通员工，系统会在 SQL 语句中强制拼接并绑定 `WHERE emp_id = :current_emp_id`，使其只能查看或操作本人记录，从根本上防止了‘越权访问’的安全风险。”

### Q11: 系统的操作审计日志 (Audit Log) 是如何实现的？它有什么用？
*   **答辩得分点**：安全审计、操作追踪、非侵入式日志、实时事件监听。
*   **回答话术**：
    > “审计日志是企业级系统的重要合规要求。我们的系统在 `audit_logs` 表中记录所有关键操作。
    > 
    > **实现方式**：我们编写了一个统一的日志写入接口（在代码中对应 `MainWindow::logAction`）。每当发生关键业务操作（如‘添加员工’、‘修改密码’、‘审批请假’、‘一键核算工资’等），控制器在调用业务 SQL 的同时，会自动调用该接口，记录：操作人ID、操作人姓名、操作类型、详细描述以及精确到秒的时间戳。
    > 
    > **实时刷新机制**：我们在日志接口中融合了**观察者模式**。在向 `audit_logs` 成功写入日志后，会立即通过 `GlobalEvents::instance()->auditRefresh()` 全局信号发出刷新广播。运行中的“审计日志”标签页接收到信号后会自动、即时地重新查询并刷新表格显示。这实现了审计流水的**实时更新与免手动刷新展示**，提升了管理员的管理体验。
    > 
    > **作用**：这为企业管理提供了‘责任追踪’能力。一旦发生敏感数据变更，管理员可以在‘审计日志’Tab 中通过时间、操作类型、操作人等多维度快速筛选，清晰还原系统的操作历史。”

### Q12: 员工请假的“起止日期”在录入时是如何防止逻辑错误的？
*   **答辩得分点**：防御性编程、防呆设计。
*   **回答话术**：
    > “我们在客户端和数据库双重层面进行了防御性编程：
    > 
    > 1.  **客户端实时限制**：
    >     在 [LeaveTab::applyLeave](file:///c:/Users/17760/Desktop/HRMS/src/tabs/LeaveTab.cpp#L76)，当用户提交请假时，系统首先比对 `startDate` 和 `endDate`。若 `startDate > endDate`，会直接弹出警告框‘开始日期不能晚于结束日期’并拦截提交。同时，日期控件的最小值限制为当前日期，防止申请过去的日期。
    > 2.  **数据格式防呆**：
    >    > 在计算请假时长时，系统直接使用 `daysTo()` 函数计算天数差，即使只请假一天，也会准确计算为 1 天，规避了由于手动输入天数带来的逻辑漏洞。”

### Q12b: 为什么要在界面设计中使用 QSS 样式表？如何避免代码中散落硬编码样式表导致的可维护性难题？
*   **答辩得分点**：QSS（Qt Style Sheets）、样式与逻辑解耦（Separation of Concerns）、动态属性（Dynamic Properties）。
*   **回答话术**：
    > “使用 QSS 可以像开发网页 CSS 一样，实现界面外观与业务逻辑的解耦，遵循了**单一职责原则**。
    > 
    > **曾遇到的问题**：系统各处按钮和控件为了好看，在 C++ 代码中散落了大量的 `setStyleSheet` 样式字符串硬编码。这导致如果要修改按钮底色，需要修改多处 C++ 代码并重新编译，给后期维护和一键换肤带来了灾难。
    > 
    > **我们的重构方案**：我们重构了系统样式，将所有的内联样式全部提取并规范化写入到全局 [style.qss](file:///c:/Users/17760/Desktop/HRMS/style.qss) 样式表中。为了解决状态切换（如保存按钮的 dark 风格、删除按钮的 danger 风格、消息铃铛的闪烁状态 active/idle），我们巧妙利用了 Qt 的**动态属性机制**。在 C++ 中只需执行 `btn->setProperty("theme", "danger")` 即可，具体的颜色、圆角、悬浮效果均交由 QSS 实现。这极大地提高了代码的可读性和系统后期美化、维护的效率。”
